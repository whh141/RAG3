# Agentic RAG 问答系统开发文档（第一阶段：技术栈审视与逐步开发）

## 1. 文档目的

本文档只解决第一阶段问题：

1. 从**第一性原理**重新审视系统目标，避免直接堆砌接口与模块。
2. 明确**技术栈是否成立**，以及哪些选型必须立即确认。
3. 明确**模块边界、代码量分布、阶段性里程碑**，保证后续开发能稳定达到 5000 行以上。
4. 约束本项目只围绕“**自主推理与检索路由机制**”展开，不把重心偏移到 Reflection / Critique / 自纠错体系。

> 结论先行：当前路线可行，但需要对前端语言、LLM 接入策略、状态图抽象、评测闭环四个点先定死，否则后续接口和数据结构容易返工。

---

## 2. 第一性原理：系统到底要解决什么问题

### 2.1 原始目标

该系统不是泛化聊天机器人，也不是复杂多智能体协作平台，而是一个用于答辩演示与问答评测的 **Agentic RAG 系统**。其最核心能力只有一条：

> 当用户提出问题时，系统能够根据问题类型，自主决定是直接回答、检索本地知识库、还是进行外部搜索，并将该路由过程以可视化形式展示出来。

### 2.2 必须满足的最小闭环

如果从“没有任何技术偏好”的角度倒推，一个可答辩的最小系统必须具备以下闭环：

1. **输入问题**：用户通过 Web 界面发起问题。
2. **判断问题类型**：系统识别该问题是校园规章、本地知识、时效性知识还是普通常识。
3. **选择路径**：
   - 走本地 RAG；
   - 或走外部搜索；
   - 或在少数情况下直接生成。
4. **收集证据**：从本地文档块或外部检索结果中取回候选证据。
5. **压缩与重排**：控制上下文长度，让答案基于更强相关证据生成。
6. **生成回答**：输出答案并给出引用来源。
7. **展示过程**：前端实时展示“系统正在做什么”。
8. **可量化评估**：用预设问答集对比传统 RAG 与 Agentic RAG。

只要这 8 个环节被严格实现，系统就已经对齐了核心目标。

### 2.3 明确不做什么

为了防止方案漂移，以下内容在第一版中**明确不做**：

- 不引入复杂 Reflection / Critique / Self-Refine 回路。
- 不设计多代理角色协作（如 Planner / Critic / Executor 全家桶）。
- 不设计过度兜底与降级链路。
- 不在第一版中追求多模态输入。
- 不把重点放在权限系统、多租户、复杂管理后台权限隔离。

这些点不是做不到，而是与“最短路径完成自主路由型 Agentic RAG”这一核心目标无关。

## 3. 技术栈审视与最终建议

## 3.1 后端：FastAPI

### 为什么成立

FastAPI 对该题目几乎是最短路径选择：

- 原生支持异步接口；
- 原生支持 WebSocket；
- 与 Pydantic、SQLAlchemy、后台任务整合成熟；
- 适合快速搭建上传、会话管理、流式问答接口；
- 毕设演示成本低，开发速度快。

### 是否需要替代

不需要。当前没有比 FastAPI 更符合“开发效率 + 异步流式输出 + Python 生态整合”的替代方案。

### 结论

**后端确定为 FastAPI，不再讨论替代方案。**

---

## 3.2 前端：Vue 3 + TypeScript + Element Plus

### 争议点：先 JavaScript，还是直接 TypeScript？

从短期上看，JavaScript 的确能更快跑通页面；但从第一性原理看，我们不是只做一个演示壳，而是要做：

- 聊天页；
- 思维链调试面板；
- 知识库管理页；
- 会话历史页；
- 评测结果展示页；
- WebSocket 流式状态同步；
- 多种消息事件类型管理。

这意味着前端一定会出现大量结构化状态：

- `ChatMessage`
- `AgentStateEvent`
- `Citation`
- `DocumentRecord`
- `EvalTask`
- `EvalResult`
- `WebSocketFrame`

如果前期使用纯 JavaScript，后期会在以下位置出现明显返工：

1. WebSocket 事件类型容易混乱；
2. 多页面共享状态难以维护；
3. 可视化调试面板的数据结构容易失控；
4. 后期补类型定义时需要重复梳理大量组件通信。

### 第一性结论

既然系统一定会增长到 5000+ 行，**一开始就使用 TypeScript 更符合总成本最小原则**。

### 结论

**前端确定为 Vue 3 + TypeScript + Element Plus。**

---

## 3.3 向量数据库：ChromaDB

### 为什么成立

ChromaDB 非常适合当前场景：

- 本地部署简单，适合毕业设计环境；
- 上手成本低；
- 与 LangChain 生态整合成熟；
- 便于快速完成文档入库、检索、删除和元数据过滤。

### 已知边界

- 大规模数据量下能力有限；
- 检索分析能力不如更重型向量库；
- 运维扩展性不是重点。

但这些边界在毕业设计场景里不是主要矛盾。

### 结论

**向量库确定为 ChromaDB。**

---

## 3.4 LLM 编排：LangChain + LangGraph

### 为什么 LangChain 需要保留

本项目需要：

- 模型调用统一封装；
- 文档加载与文本切块工具；
- 向量检索器集成；
- Prompt 模板管理。

这些基础能力 LangChain 已经足够。

### 为什么必须补充 LangGraph

你的开题报告核心亮点之一，是“**基于图结构的智能体状态管理**”。如果只使用普通链式调用，很难在架构层面体现“图状态驱动的动态路由”。

LangGraph 的价值不在于让系统更复杂，而在于：

- 可以显式定义状态 State；
- 可以显式定义节点 Nodes；
- 可以显式定义条件边 Edges；
- 非常适合展示“根据问题类型选择检索路径”的过程；
- 和 WebSocket 推送中间状态天然契合。

### 风险点

LangGraph 会带来一定学习成本，但这是对课题亮点有直接贡献的复杂度，而不是无意义复杂度。

### 结论

**智能体编排确定为 LangChain + LangGraph。**

---

## 3.5 模型接入：智谱 AI 为主，OpenAI 为对照

### 原则

模型接入要服务于两个目标：

1. 支撑中文校园问答；
2. 支撑评测对比与答辩演示。

### 为什么主模型选智谱 AI

- 中文能力适合校园规章问答；
- 成本相对可控；
- 适合高频调试；
- 便于在答辩中强调国产大模型适配性。

### 为什么保留 OpenAI

保留 OpenAI 的意义不是兜底，而是：

- 作为实验对照模型；
- 用于在评测模块中比较不同模型对路由质量和生成质量的影响；
- 支撑界面中的“模型策略切换”。

### 结论

**模型层采用策略模式：ZhipuProvider + OpenAIProvider，默认主路由使用智谱 AI。**

---

## 3.6 外部搜索：Tavily

### 为什么成立

你的题目中明确包含“时效性问题”和“自主选择是否联网”的能力。Tavily 是当前最贴近 LLM 搜索场景的方案之一：

- 返回结果更适合 LLM 消费；
- 搜索结果结构化；
- 集成成本低；
- 适合做 Agent 路由中的 Web Search Node。

### 结论

**外部搜索确定为 Tavily。**

## 4. 技术栈最终冻结建议

本阶段建议冻结为以下组合：

- **后端**：FastAPI
- **前端**：Vue 3 + TypeScript + Element Plus
- **关系数据库**：SQLite（开发与答辩默认）
- **向量数据库**：ChromaDB
- **LLM 编排**：LangChain + LangGraph
- **模型接入**：智谱 AI（默认）+ OpenAI（对照）
- **外部搜索**：Tavily
- **ORM**：SQLAlchemy
- **前端状态管理**：Pinia
- **图表展示**：ECharts

> 如无特殊原因，后续 API、Schema、目录结构、测试脚本都应基于这套冻结配置展开。

## 5. 架构设计原则

为了满足“第一性原理”与“最短路径实现”的双重要求，架构必须遵守以下约束。

### 5.1 单一主链路原则

系统主链路只能有一条：

**问题输入 → 路由判断 → 检索/搜索 → 重排序/压缩 → 生成回答 → 实时展示 → 落库记录**

不得并行再造第二套备选主链路，否则会导致逻辑与代码量失真。

### 5.2 路由优先原则

核心亮点是“自主推理与检索路由”，因此复杂度必须优先投入在：

- Router Node
- Retrieval Node
- Search Node
- Context Compression Node
- Generation Node
- State Streaming

而不是优先投入在花哨 UI 或额外管理功能。

### 5.3 状态显式化原则

所有智能体过程都必须可观测，不能只在日志里隐式存在。

因此至少要显式输出：

- 当前节点名；
- 节点输入摘要；
- 节点决策结果；
- 检索数量；
- 选中的证据来源；
- 生成完成状态。

### 5.4 数据分层原则

- **SQLite**：存系统记录、文件元数据、会话、消息、评测结果。
- **ChromaDB**：只存语义检索相关文本块。
- **LangGraph State**：只存单次问答运行时状态。

不同存储层的职责不能混用。

## 6. 模块划分与职责边界

## 6.1 前端模块

### 1）ChatView

职责：

- 展示对话消息；
- 发送用户问题；
- 接收流式回答；
- 渲染引用来源；
- 管理当前会话。

### 2）CoTDebugger

职责：

- 展示每个节点状态；
- 按时间线显示路由过程；
- 支持查看检索命中和最终决策。

注意：这里展示的是**系统推理过程的可视化摘要**，不是泄露模型完整私有思维链。

### 3）KBManager

职责：

- 上传文档；
- 查看处理状态；
- 查看切片数量；
- 删除文档；
- 展示向量化进度。

### 4）HistoryView

职责：

- 管理历史会话；
- 读取历史问答记录；
- 支持按时间回溯。

### 5）EvaluationView

职责：

- 启动评测任务；
- 展示传统 RAG vs Agentic RAG 对比结果；
- 展示准确率、引用命中率、响应时间、路径选择正确率等图表。

---

## 6.2 后端模块

### 1）API Router 层

职责：

- 提供 RESTful 接口；
- 提供 WebSocket 接口；
- 做入参校验和响应组装。

### 2）Document Pipeline 层

职责：

- 文件保存；
- 文档解析；
- 文本清洗；
- 分块策略执行；
- 向量化入库；
- 元数据写入 SQLite。

### 3）Agent Core 层

职责：

- 定义 Graph State；
- 定义 Router Node / Local RAG Node / Web Search Node / Rerank Node / Generate Node；
- 控制节点跳转；
- 统一事件推送。

### 4）Persistence 层

职责：

- 管理 SQLite 模型；
- 管理 ChromaDB collection；
- 提供仓储方法。

### 5）Evaluation Pipeline 层

职责：

- 批量读取测试集；
- 分别运行传统 RAG 与 Agentic RAG；
- 记录指标；
- 输出 JSON、CSV 与图表。

## 7. 代码量分布审视

目标不是“机械凑够 5000 行”，而是确保每一部分代码都对应真实复杂度。

## 7.1 推荐分布

| 模块 | 目标代码量 | 说明 |
| --- | ---: | --- |
| 前端界面与状态管理 | 1800 - 2200 行 | 聊天、调试面板、知识库、评测视图、状态管理 |
| 后端 API 与基础设施 | 1300 - 1700 行 | 路由、数据库模型、服务层、文档处理流水线 |
| Agent 核心与图工作流 | 1000 - 1400 行 | State、Nodes、条件边、上下文压缩、检索融合 |
| 自动化评测模块 | 600 - 900 行 | 数据集读取、执行器、打分器、报表生成 |
| 测试、脚本与配置 | 300 - 500 行 | 单元测试、集成测试、初始化脚本 |
| **总计** | **5000+ 行** | 满足工程量要求 |

## 7.2 为什么这个分布合理

- 前端并不只是一个聊天壳，还承载可视化与评测展示，因此 2000 行左右是合理的。
- 后端需要承担文档处理与持久化，天然会形成较多服务代码。
- Agent 核心虽然是亮点，但真正代码量不会无限膨胀，1000+ 行是健康范围。
- 评测模块是答辩抓手，必须独立成层，否则系统难以证明效果。

## 8. 逐步开发路线（必须按顺序推进）

以下顺序是为了避免返工与逻辑漂移。

## 阶段 0：冻结技术栈与对象边界

本阶段产物：

- 技术栈确认；
- 核心模块确认；
- 代码量分布确认；
- 术语统一。

验收标准：

- 能明确回答“为什么要用 LangGraph”；
- 能明确回答“前端为什么直接用 TypeScript”；
- 能明确回答“系统核心亮点不是反思纠错，而是路由机制”。

## 阶段 1：定义核心数据结构与目录骨架

本阶段产物：

- SQLAlchemy 模型；
- Pydantic Schema；
- LangGraph State Schema；
- 项目目录初始化。

验收标准：

- `Document`、`ChatSession`、`ChatMessage`、`EvalRun`、`EvalRecord` 等对象边界明确；
- WebSocket 事件格式明确；
- 前后端共享的字段命名统一。

## 阶段 2：完成知识库处理闭环

本阶段产物：

- 文档上传；
- 文本解析；
- 切块；
- 向量化；
- ChromaDB 入库；
- 文档列表查询；
- 文档删除。

验收标准：

- 上传一个校园规章 PDF 后，系统能看到处理状态与切片数量；
- 删除文档后，SQLite 与 ChromaDB 状态一致。

## 阶段 3：完成基础问答闭环（先不加 Agent 路由）

本阶段产物：

- 普通 RAG 问答；
- 检索结果引用；
- 聊天记录存储；
- WebSocket 流式输出。

验收标准：

- 用户能基于本地知识库完成一轮正常问答；
- 前端能看到流式答案与引用来源。

## 阶段 4：加入 Agent 路由与 LangGraph

本阶段产物：

- Router Node；
- Local RAG Node；
- Web Search Node；
- Rerank / Compression Node；
- Final Generation Node；
- 状态流推送。

验收标准：

- 系统能对问题进行路径选择；
- 前端能看到节点流转；
- 回答能明确引用本地文档或外部搜索来源。

## 阶段 5：补齐评测体系

本阶段产物：

- 问答测试集；
- 基线 RAG 执行器；
- Agentic RAG 执行器；
- 指标计算；
- 图表输出。

验收标准：

- 能批量运行至少 100 个问题；
- 输出准确率、延迟、引用命中率、路由正确率；
- 可以生成答辩演示用图表。

## 阶段 6：前端增强与论文答辩展示优化

本阶段产物：

- 调试面板优化；
- 评测可视化；
- 会话管理完善；
- 系统说明页。

验收标准：

- 页面结构完整；
- 演示路径清晰；
- 能展示系统“为什么做出该路由决策”。

## 10. 本阶段需要确认的关键决策

在进入 API 和 Schema 设计之前，建议你先确认以下四件事：

### 决策 1：前端是否直接上 TypeScript

**建议答案：是。**

原因：系统状态复杂，后期补类型的成本会更高。

### 决策 2：是否将 LangGraph 设为必选而非可选

**建议答案：是。**

原因：这是题目亮点的核心承载层。

### 决策 3：默认主模型是否固定为智谱 AI

**建议答案：是。**

原因：中文场景与成本都更适合。

### 决策 4：第一版是否严格聚焦“路由 + 检索 + 可视化 + 评测”

**建议答案：是。**

原因：这是最短路径闭环，不应被额外业务稀释。

## 12. 本阶段最终结论

基于第一性原理审视后，可以得出以下结论：

1. 你的题目本质是一个 **以动态路由为核心的 Agentic RAG 问答系统**；
2. 技术栈整体成立，但前端应直接切到 **Vue 3 + TypeScript**；
3. **LangGraph 必须保留**，因为它是“图结构状态管理”最直接的实现载体；
4. 代码量 5000+ 的目标是合理的，但必须通过“前端可视化 + 后端流水线 + Agent 路由 + 评测体系”共同支撑；
5. 开发必须按阶段推进，不能在未冻结技术栈前就开始大面积写接口和数据结构。

至此，第一阶段文档完成。
